# NNN GAMES 랜딩 페이지 개발 가이드라인
회사 기본 정보와 프로젝트/업적을 명확히 알리는 랜딩 페이지를 대상으로, 정적 웹사이트(HTML/CSS/JS) 구조를 유지하면서도 확장성·보안·운영 편의를 높이는 것을 목표로 한다.

## 1. 범위와 원칙
- 목적: 회사 기본 정보, 프로젝트 레퍼런스, 업적을 명확히 전달하고 CTA(문의/프로젝트 상세/플레이 링크)로 전환 유도.
- 품질 우선순위: 접근성 ≥ 성능 ≥ 일관된 UI/브랜드 ≥ SEO/측정 ≥ 개발 생산성.
- 변경 관리: 모든 수정은 PR + 코드리뷰 1인 이상. PR 템플릿에 변경 요약, 테스트 결과, 리스크를 명시하고 스크린샷/미리보기 링크를 첨부.

## 2. 정보 구조 & 코드베이스
- 정적 페이지: `index.html`, `about.html`, `projects.html`, 개별 프로젝트 상세(`get-train.html` 등), `contact.html`.
- 데이터 소스: `js/projects-data.js`에서 프로젝트 메타를 관리하고 `js/project-renderer.js`로 카드/배지를 생성.
- 다국어: `js/i18n.js` + `data-key` 속성. KO/EN/JA 유지, 새 텍스트는 세 언어 모두 추가.
- 공통 자산: `css/style.css`, `images/` 썸네일·갤러리, `js/main.js`(네비/애니메이션/지연 로딩).

## 3. 콘텐츠 관리 규칙
- 프로젝트 추가/수정 시 `projects-data.js`와 `i18n.js` 번역 키를 동시에 반영, 필요하면 상세 HTML도 갱신.
- 이미지: 웹용 압축(JPEG/WebP), 1~2MB 이하, 가로 1200px 권장. 의미 있는 `alt` 필수.
- 텍스트: 제품·클라이언트 명칭 표준화(공식 표기 사용), 날짜는 ISO(YYYY-MM 또는 YYYY-Qn), CTA 문구는 액션 중심으로 작성.

## 4. UI/UX 가이드
- 반응형: 768px 이하 모바일 내비 토글 정상 동작 확인. 터치 타겟 44px 이상.
- 내비/CTA: Hero CTA는 우선 홍보 프로젝트 상세로 연결, 모든 주요 CTA는 트래킹 가능하도록 별도 클래스/데이터 속성 부여.
- 상태/플랫폼 배지: `category/status` 값이 데이터와 일치하도록 유지. 새 플랫폼은 `ProjectRenderer.getPlatformInfo` 확장.

## 5. 국제화(i18n)
- `document.documentElement.lang`를 언어 전환 시 업데이트하고 `localStorage`에 저장.
- 새 콘텐츠는 `data-key`를 부여하고 KO/EN/JA 모두 번역. 누락 키가 없도록 린트/리뷰에서 확인.
- 날짜/단위는 언어 중립적으로 표기하거나 언어별 포맷을 명시.

## 6. 접근성
- 시맨틱 마크업 사용: 제목 계층(H1-H3), 리스트, 버튼/링크 구분.
- 키보드 내비게이션: 포커스 가능한 요소, 스킵 링크(헤더 바로 아래) 추가 고려.
- 대비: WCAG AA 대비 충족, 배지/버튼 컬러 대비 점검.
- 이미지/미디어: 의미 있는 `alt`, 데코 이미지는 빈 `alt`. iframe/video에는 `title` 부여.

## 7. 성능 & 프런트엔드 품질
- 이미지 지연 로딩 유지, 필요 시 `loading="lazy"` 병행.
- 정적 자원 최소화: CSS/JS 경량 유지, 불필요한 라이브러리 금지.
- 캐싱: 정적 파일은 캐시 가능하게, 변경 시 파일명/쿼리로 캐시 무효화.
- 측정: Lighthouse(Performance/Accessibility/Best Practices/SEO) 90+ 목표. 주요 페이지(홈/프로젝트 상세) 정기 체크.

## 8. SEO & 메타
- 각 페이지에 고유한 `<title>`, `<meta name="description">`, OG/Twitter 카드 메타 추가. Hero/CTA 주요 키워드 반영.
- `sitemap.xml`, `robots.txt` 제공 권장. CNAME 도메인 일관성 확인.
- 링크 체크: 외부/내부 깨진 링크 없도록 정기 검증. CTA와 외부 플레이 링크에 UTM/추적 파라미터 필요 시 명시적으로 관리.

## 9. 보안/개인정보
- 사용자 입력 폼 추가 시: 최소 검증, 스팸 방지(CAPTCHA/토큰), 서버 전송 시 HTTPS 강제.
- 서드파티 스크립트 최소화, 무결성(SRI) 적용 가능 시 적용.
- PII 수집 최소화. 이메일/주소 외 정보 수집 시 별도 개인정보 고지 필요.

## 10. 관측/분석
- 웹 분석 도입 시: 개인정보 최소화 설정(IP anonymize 등), 이벤트 스키마(CTA 클릭, 언어 전환, 외부 링크 클릭) 정의 후 적용.
- 장애 모니터링이 필요한 경우 경량 에러 로깅(Sentry 등) 추가하되 번들 크기 영향 검토. 외부 스크립트는 SRI/nonce 적용 가능 여부 검토.

## 11. 테스트 & 검증 체크리스트
- 기능: 언어 전환, 모바일 내비 토글, 프로젝트 카드 렌더, CTA 링크 동작, 이미지 로딩.
- 접근성: 키보드 탭 이동, 포커스 링 확인, 스크린리더 주요 섹션 읽기.
- 성능/SEO: Lighthouse 90+ (모바일/데스크톱), 메타 태그/OG 확인.
- 링크/리소스: 404 없는지 크롤러/스캐너로 점검.

## 12. 배포/운영
- 정적 호스팅(GitHub Pages 등) 기준: 빌드 없이 배포하므로 수동 검증 필수.
- CNAME/HTTPS 확인, 캐시 무효화 후 주요 페이지 스팟 체크.
- 롤백 전략: 이전 릴리스(태그/브랜치)로 신속 교체 가능하게 태깅.

## 13. 변경 프로세스
- PR 템플릿 권장 필드: 변경 요약, 테스트(수동 체크 항목 포함), 스크린샷/미리보기 링크, 릴리스 노트 영향.
- 리뷰 기준: i18n 키 누락 여부, 접근성 퇴행, 링크/CTA 정확도, 데이터 스키마 준수, 이미지 최적화 여부.

## 14. 자주 추가되는 확장 포인트 (미리 정의)
- 프로젝트 필터/정렬 UI 연결 시 `ProjectRenderer.filterByCategory/Status` 활용.
- 새 플랫폼/카테고리 추가 시 `ProjectRenderer.getPlatformInfo`와 CSS 배지 클래스 확장.
- SEO 강화 시 각 페이지별 OG 이미지/설명 개별화, 구조화 데이터(Organization, Breadcrumb) 검토.

## 15. 대규모 프로젝트(수백 개) 관리 가이드
- 데이터 소스 분리: `projects-data.js`를 JSON 분할(예: 카테고리/연도별) 후 빌드 시 병합, 프런트에는 경량 index만 로드.
- 리스트 최적화: 기본 뷰는 페이징 또는 무한 스크롤 + 필터/검색(텍스트, 카테고리, 상태). 카드에 필수 필드만 포함하고 상세는 지연 로드.
- 정적 빌드 전략: 정적 HTML은 공통 셸 + 클라이언트 렌더; 대량 데이터는 JSON fetch + 캐시. 필요 시 정적 검색 인덱스(예: prebuilt JSON) 사용.
- 이미지/자산: 썸네일/갤러리 분리, WebP 우선, srcset/`loading="lazy"` 적용, 파일명 규칙(슬러그_사이즈.ext) 유지.
- 스키마: 필수 키(id, slug, title{ko,en,ja}, description{...}, category, status, platform, launchDate, metrics, links{play, video, article}, thumbnail). 확장 필드는 optional로 관리.
- URL/슬러그: `projects/<slug>.html` 규칙 유지, slug는 고유·소문자·하이픈.
- i18n: 데이터 레벨에서 다국어 필드 병행 관리, 새 키 추가 시 세 언어 동시 추가.
- 퍼포먼스: 초기 로드 데이터 최소화, 필터/검색 시 debounce, 큰 리스트는 가상 스크롤 고려.
- SEO: 주요 대표 프로젝트는 개별 정적 페이지 유지, 대량 리스트는 카테고리/태그 페이지로 내부 링크 구조 확보.
